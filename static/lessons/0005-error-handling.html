Error Handling in SvelteKit

+error.svelte, error(), and handleError

What you'll learn

  • The difference between expected errors and unexpected errors
  • How +error.svelte works as a scoped error boundary
  • How to throw HTTP errors from load functions using error()
  • How to read $page.error and $page.status inside an error page
  • How handleError in hooks.server.ts lets you centralise error logging

Part 1 — What happens today when something breaks?

Your app has load functions that query Prisma, form actions that validate input, and a protected route group that redirects unauthenticated users. But try visiting /dashboard/some-page-that-doesnt-exist — you'll get a raw SvelteKit 404 page that looks nothing like your app.

Or imagine your Prisma query throws because the database is down. The user sees a generic 500 — no context, no "go back" link, no branding. That's what we're fixing today.

Coming from Nuxt

Nuxt has a single ~/error.vue that catches everything globally. SvelteKit is more granular — each segment of the route tree can have its own +error.svelte, like nested catch blocks. A dashboard error page stays in the dashboard shell; a root error page is the fallback for everything else.

Part 2 — Expected vs. unexpected errors

SvelteKit draws a firm line between two types of errors. Understanding this distinction is the key to using the system correctly.

Type How it's created What the user sees Goes through handleError?
Expected You throw error(404, 'Not found') Your +error.svelte with the message No — you already know about it
Unexpected Any other thrown error (Prisma crash, null ref, etc.) Your +error.svelte with a generic message Yes — so you can log it

Expected errors are things you chose to reject: a missing resource, a forbidden action, a bad ID in the URL. You use SvelteKit's error() helper to throw them.

Unexpected errors are bugs: database down, null pointer, bad env var. SvelteKit catches these too and shows your error page, but it hides the message from the user (it could contain sensitive info) and instead calls your handleError hook so you can log it.

Important distinction

fail() is NOT the same as error(). You already use fail() in form actions to return validation errors back to the form — the user stays on the same page. error() is for hard failures that should render the error page instead.

Throwing an expected error

Import error from @sveltejs/kit and call it anywhere in a load function or form action. The first argument is the HTTP status code, the second is the message.

src/routes/notes/[id]/+page.server.ts
import { error } from '@sveltejs/kit';
import prisma from '$lib/server/db';

export const load = async ({ params }) => {
  const note = await prisma.note.findUnique({
    where: { id: Number(params.id) }
  });

  if (!note) {
    error(404, 'Note not found');  // throws — stops execution
  }

  return { note };
};

Calling error() throws immediately — nothing after it runs. SvelteKit catches it, sets the HTTP response to 404, and renders the nearest +error.svelte.

Part 3 — +error.svelte: the error boundary

+error.svelte is a special component SvelteKit renders when a load function or action throws. It replaces the normal +page.svelte content but inherits the surrounding layout — so your nav, header, and shell stay in place.

The two stores available inside +error.svelte

You access error information via the $page store:

src/routes/+error.svelte
<script>
  import { page } from '$app/stores';
</script>

<div class="error-page">
  <h1>{$page.status}</h1>
  <p>{$page.error?.message}</p>
  <a href="/">Go home</a>
</div>
  • $page.status — the HTTP status code (404, 500, etc.)
  • $page.error.message — the message from error(404, 'Note not found')
Unexpected errors hide the message

For unexpected errors (unhandled exceptions), $page.error.message is always "Internal Error" regardless of what was thrown. This is intentional — SvelteKit won't leak stack traces or database errors to the browser.

Where does +error.svelte go?

SvelteKit finds the nearest +error.svelte by walking up the route tree. This is where the power lies.

src/routes/ ├── +error.svelte ← catches everything (root fallback) ├── +layout.svelte ├── (protected)/ │ ├── +error.svelte ← catches errors inside protected routes only │ ├── +layout.server.ts │ └── dashboard/ │ └── +page.svelte └── examples/ └── notes/ └── +page.svelte

If the dashboard load throws, SvelteKit renders (protected)/+error.svelte — inside the protected layout shell. The root error page never fires for dashboard errors.

If a route outside (protected) throws, it bubbles up to +error.svelte at the root.

Why scoped errors matter

Your dashboard has a nav sidebar. If the load throws and the root error page renders, the sidebar disappears — confusing. With a scoped (protected)/+error.svelte, the sidebar stays visible and the error message appears inside it, where the user expects.

Part 4 — handleError: catching the unexpected

You already have hooks.server.ts with a handle function. The same file can export a second hook: handleError. It fires whenever an unexpected error escapes a load function or action on the server.

This is where you'd log to a service like Sentry, write to a log file, or simply console.error in development. The user never sees what you log here.

src/hooks.server.ts
import type { Handle, HandleServerError } from '@sveltejs/kit';

// your existing handle hook...
export const handle: Handle = async ({ event, resolve }) => {
  // ... JWT check, locals.user, etc.
  return resolve(event);
};

// NEW: catches unhandled server errors
export const handleError: HandleServerError = ({ error, event }) => {
  console.error('[Server Error]', {
    path: event.url.pathname,
    error
  });

  // The returned object becomes $page.error
  return {
    message: 'Something went wrong. Please try again.'
  };
};

The object you return from handleError becomes the shape of $page.error in your error page. The default is { message: 'Internal Error' }. You can return a custom message (safe to show) while keeping the real error in your logs.

TypeScript: typing the error shape

If you extend what handleError returns, update app.d.ts so the type is known everywhere: 

// src/app.d.ts
declare global {
  namespace App {
    interface Error {
      message: string;
    }
    // ... Locals, PageData, etc.
  }
}

The full picture

Here's how an error in a load function flows through the system:

📂 Load function runs (e.g. +layout.server.ts)
↓
💥 An error is thrown — either error(404, ...) or an unhandled exception
↓
🎣 Unexpected only: handleError fires in hooks.server.ts — log it here
↓
🖼 Nearest +error.svelte renders inside the surrounding layout shell
↓
📖 Component reads $page.status and $page.error.message
