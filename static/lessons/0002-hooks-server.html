Lesson 2 · SvelteKit

hooks.server.ts
Centralized Middleware

Move auth out of every load function. Run logic once, for every request.

⏱ ~20 min 📋 Quiz + 4 exercises 🎯 Tied to: auth, protected routes
Coming from Nuxt
hooks.server.ts is SvelteKit's equivalent of Nuxt server middleware (server/middleware/*.ts). It runs on every request before your routes handle anything. The event.locals object is the same idea as Nuxt's event.context — a place to attach request-scoped data for downstream handlers.

The Problem This Solves

Right now, your /admin route reads the JWT cookie and verifies it inside its own load function. Every protected route has to do this. That's a problem:

  • Duplicated auth logic across every protected +page.server.ts
  • If you change how auth works, you update every file
  • No single place to reason about "who is allowed where"

hooks.server.ts fixes this. You verify the token once, attach the user to the request pipeline, and every load function simply reads locals.user.

The Request Pipeline

Every incoming request in SvelteKit flows through this pipeline:

Browser Request
→
handle()
hooks.server.ts
→
resolve(event)
→
load() / action()
+page.server.ts
→
Response

The handle hook is a gate. You can inspect and modify the request before it reaches your routes, or short-circuit it entirely (e.g., redirect an unauthenticated user before the load function even runs).

The handle Hook

Create the file src/hooks.server.ts and export a function called handle:

src/hooks.server.ts
import type { Handle } from '@sveltejs/kit';

export const handle: Handle = async ({ event, resolve }) => {
  // pre-processing: runs before the load function
  const response = await resolve(event);
  // post-processing: runs after the load function
  return response;
};

Two parameters:

  • event — the full request context: event.url, event.cookies, event.request, event.locals, and more
  • resolve — calls the matched route's load function and renders the response. You must call this and return what it gives you — or return your own Response instead
Key rule
If you don't call resolve(event), SvelteKit never reaches your route. This is how you intercept and redirect — return a redirect() before calling resolve.

event.locals — Passing Data Down the Pipeline

event.locals is a plain object that persists for the lifetime of a single request. You write to it in handle, and read from it in load functions and +server.ts routes.

Think of it as the request's "shared memory" — only for server-side code, only for one request.

src/hooks.server.ts — writing to locals
event.locals.user = verifiedPayload;  // set it here
src/routes/admin/+page.server.ts — reading locals
export const load: PageServerLoad = async ({ locals }) => {
  const user = locals.user;  // read it here — no cookie/JWT needed
};

Typing event.locals with app.d.ts

TypeScript doesn't know what you'll put in event.locals unless you declare it. SvelteKit provides a global App namespace for this — you extend it in src/app.d.ts:

src/app.d.ts — declare the shape of event.locals
import type { AuthTokenPayload } from '$lib/types/auth';

declare global {
  namespace App {
    interface Locals {
      user: AuthTokenPayload | null;
    }
  }
}

export {};

After this, event.locals.user is fully typed everywhere — in handle, in load functions, and in +server.ts routes. Official reference →

Why store AuthTokenPayload instead of SessionUser?
AuthTokenPayload includes userId — which you'll need when you start making database queries in load functions (e.g., fetching a user's records). SessionUser only has display info. Store the richer type in locals; project it to a narrower type when you return it from load.

The Auth Pattern: Verify Once, Use Everywhere

Here's the canonical pattern for JWT auth in SvelteKit hooks:

src/hooks.server.ts
import type { Handle } from '@sveltejs/kit';
import { verifyToken } from '$lib/server/auth';

export const handle: Handle = async ({ event, resolve }) => {
  const token = event.cookies.get('token');
  event.locals.user = token ? await verifyToken(token) : null;
  return resolve(event);
};

That's it. Now every load function, every server route, and every form action has access to event.locals.user without touching a cookie or running a JWT verify call.

Compare what your /admin load function looks like before and after:

✗ Before — repeated auth logic
export const load = async ({ cookies }) => {
  const token = cookies.get('token');
  if (!token) throw redirect(303, '/');
  const payload = await verifyToken(token);
  if (!payload) throw redirect(303, '/');
  return { user: { email: payload.email, name: payload.name } };
};
✓ After — locals already set
export const load = async ({ locals }) => {
  if (!locals.user) throw redirect(303, '/');
  const { email, name } = locals.user;
  return { user: { email, name } };
};

Route Guarding: Two Approaches

Approach A — Guard in the load function (simple, explicit)

The hook sets locals.user. The load function decides whether to redirect:

+page.server.ts (any protected route)
export const load: PageServerLoad = async ({ locals }) => {
  if (!locals.user) throw redirect(303, '/');
  // ... rest of load
};

Best for most apps. Each route remains self-documenting — you can see at a glance which routes are protected.

Approach B — Guard in handle (centralized, declarative)

The hook itself redirects unauthenticated requests to protected paths:

src/hooks.server.ts — with route guard
const protectedPaths = ['/admin', '/dashboard'];

export const handle: Handle = async ({ event, resolve }) => {
  const token = event.cookies.get('token');
  event.locals.user = token ? await verifyToken(token) : null;

  const isProtected = protectedPaths.some(
    (path) => event.url.pathname.startsWith(path)
  );

  if (isProtected && !event.locals.user) {
    throw redirect(303, '/');
  }

  return resolve(event);
};

Best when you have many protected routes or want a single authoritative list. The redirect happens before the load function even starts — so no wasted work.

Composing Multiple Hooks with sequence()

As your app grows, you may want separate handle functions for auth, logging, or i18n. SvelteKit provides a sequence() helper that chains them:

src/hooks.server.ts
import { sequence } from '@sveltejs/kit/hooks';
import { verifyToken } from '$lib/server/auth';
import type { Handle } from '@sveltejs/kit';

const auth: Handle = async ({ event, resolve }) => {
  const token = event.cookies.get('token');
  event.locals.user = token ? await verifyToken(token) : null;
  return resolve(event);
};

const logger: Handle = async ({ event, resolve }) => {
  console.log(`${event.request.method} ${event.url.pathname}`);
  return resolve(event);
};

export const handle = sequence(logger, auth);
// logger runs first, then auth — order matters

Official handle docs →

Bonus: handleError

There's a second common hook: handleError. It runs whenever an unexpected error escapes a load function or action — giving you a chance to log it and return a safe user-facing message:

src/hooks.server.ts
export const handleError = ({ error, event }) => {
  console.error('Unhandled error:', error);
  return { message: 'Something went wrong.' };
};

The return value becomes $page.error in your error page. Official handleError docs →

A { request, response }
B { req, res }
C { event, resolve }
D { context, next }
That's a different framework's convention. SvelteKit uses its own naming — event is the full request context, and resolve is the function that calls your route's load function.
Correct! event gives you everything about the request (URL, cookies, locals…) and resolve is how you delegate to the matched route. You must call and return resolve(event) unless you're short-circuiting entirely.

A event.data
B event.locals

event.context is Nuxt/Nitro's equivalent — but SvelteKit uses event.locals. No event.data or event.store exist.
Correct! event.locals is the request-scoped shared memory. Write to it in handle, read from it in load, +server.ts routes, and form actions — all within the same request lifecycle.

A src/lib/types/locals.ts
B tsconfig.json

D src/app.d.ts

5 In handle, you return a redirect(303, '/') without calling resolve(event). What happens?
A The redirect is queued; the load function still runs
B SvelteKit throws an error — you must call resolve first
C The load function never runs; the redirect is sent immediately
D The redirect is ignored and the route loads normally
Remember: resolve(event) is what invokes the route's load function. If you return before calling it, SvelteKit sends your response (the redirect) and the route never runs. This is why hook-level guards are so efficient.
Correct! The hook is a gate. Returning early (with a redirect, a custom Response, or an error) prevents the load function from ever running — no wasted database calls or auth checks on the route side.
6 You have three separate Handle functions: auth, logger, i18n. How do you export them all as the single handle?

C Use the sequence() helper from @sveltejs/kit/hooks

SvelteKit only looks for one handle export, and Handle functions aren't Promises so .then() doesn't apply. The sequence() helper was made exactly for this — it composes Handle functions and runs them in order.
Correct! sequence(logger, auth, i18n) runs them left-to-right. Each handler calls resolve which passes control to the next. If any handler returns early (redirect, error), the chain stops there.

Hands-On Exercises

These exercises modify your real project. By the end, your app will have a proper auth middleware layer.

Exercise 1 Create hooks.server.ts with a request logger

Start simple. Create the file and confirm the hook runs on every request.

import type { Handle } from '@sveltejs/kit';

export const handle: Handle = async ({ event, resolve }) => {
  console.log(`${event.request.method} ${event.url.pathname}`);
  return resolve(event);
};
✓ Verified when: Terminal shows request logs for every navigation. Note that SvelteKit's internal requests (like HMR) may also appear — that's expected.
Exercise 2 Create src/app.d.ts and type event.locals

Add TypeScript awareness of locals.user across the entire app.

import type { AuthTokenPayload } from '$lib/types/auth';

declare global {
  namespace App {
    interface Locals {
      user: AuthTokenPayload | null;
    }
  }
}

export {};
✓ Verified when: TypeScript gives you autocomplete on event.locals.user and shows no type errors. If you see a "property 'user' does not exist" error, make sure app.d.ts is at src/app.d.ts (not inside lib/).
Exercise 3 Add JWT verification to the hook

Expand the hook to verify the auth cookie and populate locals.user on every request.

import type { Handle } from '@sveltejs/kit';
import { verifyToken } from '$lib/server/auth';

export const handle: Handle = async ({ event, resolve }) => {
  const token = event.cookies.get('token');
  event.locals.user = token ? await verifyToken(token) : null;
  return resolve(event);
};
✓ Verified when: When logged in, the terminal shows your user object (email, name, userId). When logged out, it shows null.
Exercise 4 Simplify /admin to use locals.user

Remove the duplicated auth logic from the admin load function — hooks.server.ts has already done the work.

import { redirect } from '@sveltejs/kit';
import type { PageServerLoad } from './$types';

export const load: PageServerLoad = async ({ locals }) => {
  if (!locals.user) throw redirect(303, '/');

  const { email, name } = locals.user;
  return { user: { email, name } };
};
✓ Verified when: The admin page behaves identically to before — unauthenticated users are redirected, authenticated users see their data — but the load function is now 5 lines instead of 14.
