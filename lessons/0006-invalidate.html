Lesson 6 · SvelteKit

invalidate — On-Demand Data Refresh

Stop manually managing state with onMount + fetch. Let SvelteKit re-run your load functions when you need fresh data.

The problem: load functions only run on navigation

SvelteKit load functions are brilliant — they run on the server, prepare data for the page, and the component just reads data.whatever. But they run at a specific moment: when the user navigates to the route.

So what happens after you mutate data client-side — say, upload a file via fetch — and you want the page list to refresh? The load function doesn't re-run automatically. You need to tell SvelteKit to re-run it.

If you've worked with Nuxt, you've hit this with useAsyncData — you call refreshNuxtData('key') after a mutation. SvelteKit's equivalent is invalidate().

Key idea
invalidate(url) tells SvelteKit: "re-run any load function that fetched this URL." The page data updates reactively — no manual state, no onMount.

How SvelteKit tracks what a load function depends on

This is the clever part. When SvelteKit runs a server load function, it gives that function a special, instrumented fetch — not the raw browser fetch, but one SvelteKit controls. Every URL you call inside that load function is recorded as a dependency.

load({ fetch }) { ← special fetch, not browser fetch
const res = await fetch('/api/images'); ← SvelteKit records this URL
return { images }; ← dependency: '/api/images'
}
 
Client: await invalidate('/api/images') ← SvelteKit re-runs the load above

This means you don't need to name your data with a key (like Nuxt's useAsyncData('key', fn)). The URL is the key. If your load function fetched that URL, SvelteKit knows to re-run it.

The API

invalidate(url) — targeted refresh

Re-runs any load function (on the current page) that fetched the given URL. Everything else stays untouched.

import { invalidate } from '$app/navigation';

// After uploading an image:
await invalidate('/api/images');
// SvelteKit re-runs any load that fetched '/api/images'
// The page's `data.images` updates automatically.

invalidateAll() — nuclear option

Re-runs all load functions for the current page. Useful when you're not sure which URL to target, or when multiple things changed at once.

import { invalidateAll } from '$app/navigation';

await invalidateAll();  // re-runs every load function on this page
Watch out
invalidateAll() re-runs layout load functions too, not just the page load. On a page with several layout loads, this causes extra server work. Prefer invalidate(url) when you know the URL.

Custom invalidation keys

You can also invalidate on a custom string key instead of a URL. In the load function, call depends('myapp:images') to register the dependency. Then invalidate('myapp:images') to trigger it.

// +page.server.ts
export const load = async ({ fetch, depends }) => {
  depends('myapp:images');          ← register a custom dependency key
  const res = await fetch('/api/images');
  const { images } = await res.json();
  return { images };
};

// +page.svelte
await invalidate('myapp:images');   ← trigger by key, not URL

Custom keys are useful when the URL is unpredictable or when you want the dependency to be explicit and named, not tied to an API path.

Before and after: the admin image gallery

Your current admin page uses onMount + fetch + manual state for the image gallery. Here's how that pattern compares to the SvelteKit-native approach.

Before — manual fetch + state
// +page.svelte (no load function for images)

let images = $state([]);

onMount(async () => {
  await fetchImages();
});

async function fetchImages() {
  const res = await fetch('/api/images');
  const d = await res.json();
  images = d.images ?? [];
}

async function uploadImage() {
  // ... upload ...
  await fetchImages(); ← manual re-fetch
}
After — load function + invalidate
// +page.server.ts
export const load = async ({ fetch, locals }) => {
  // ... auth check ...
  const res = await fetch('/api/images');
  const { images } = await res.json();
  return { user, images };
};

// +page.svelte
import { invalidate } from '$app/navigation';
let { data } = $props();
// data.images comes from the load function

async function uploadImage() {
  // ... upload ...
  await invalidate('/api/images'); ← done
}

The "after" version has no local image state, no onMount, and no manual tracking of whether images are loaded. The load function owns the data; invalidate refreshes it.

Nuxt comparison
In Nuxt you'd use const { data } = useAsyncData('images', () => $fetch('/api/images')) and call refreshNuxtData('images') after upload. SvelteKit's equivalent is a server load function using its instrumented fetch, then invalidate('/api/images'). The Nuxt approach requires a string key; SvelteKit uses the URL itself as the key — less ceremony.
Quick check — 3 questions

1. Why doesn't a SvelteKit load function re-run automatically after you upload a file via fetch?

Load functions run when you navigate to a page, and again when SvelteKit is told to re-run them (via invalidation or navigation). A raw fetch in a component doesn't trigger either — the load function has no idea you made a request from the component.

2. Your load function calls fetch('/api/users') and fetch('/api/posts'). You call invalidate('/api/users'). What happens?

Invalidation is per-load-function, not per-URL. When SvelteKit decides a load function needs to re-run, it runs the whole function from scratch — all its fetch calls execute again. If you needed to separate them, you'd split into two load functions (one in the layout, one in the page).

3. When would you prefer depends('myapp:images') + invalidate('myapp:images') over URL-based invalidation?

SvelteKit can only track dependency URLs automatically when the load function uses its instrumented fetch. If your load function queries Prisma directly (no URL), there's no URL to track. Use depends('myapp:notes') to register a custom key, then invalidate('myapp:notes') to trigger it. This also makes the dependency explicit and greppable in your codebase.
Hands-on exercise

Refactor the admin image gallery

Your admin page (src/routes/admin/+page.svelte) currently loads images in onMount and refreshes manually after upload. You'll move that logic to the server load function and use invalidate to refresh.

1

Add image loading to the existing server load function.

Open src/routes/admin/+page.server.ts. It already returns { user }. Add a fetch('/api/images') call and return images alongside it.

The load function already receives { fetch } — add it to the destructured parameter. This is SvelteKit's instrumented fetch, not the browser one. 
export const load: PageServerLoad = async ({ locals, fetch }) => {
  // existing auth check stays
  if (!locals.user) throw redirect(303, '/');

  const res = await fetch('/api/images');
  const { images } = await res.json();

  return { user: locals.user, images: images ?? [] };
};
2

Remove onMount and the local images state from the component.

In src/routes/admin/+page.svelte, delete the images = $state([]), the fetchImages function, and the onMount block. You no longer need the onMount import either.

The data.images from the load function replaces all of it.

3

Replace the manual re-fetch with invalidate.

In the uploadImage function, instead of calling await fetchImages(), call await invalidate('/api/images').

import { invalidate } from '$app/navigation';

async function uploadImage(event: Event) {
  // ... existing upload code ...

  if (response.ok) {
    uploadMessage = 'Upload succeeded.';
    selectedFile = null;
    await invalidate('/api/images');  ← replaces fetchImages()
  }
}
4

Update the template to use data.images.

In the template, change {#each images as image} to {#each data.images as image}. The check for empty state becomes {#if data.images.length}.

5

Test it.

  • Navigate to /admin — images should load immediately (no onMount delay).
  • Upload a new image — the gallery should refresh automatically without a page reload.
  • Open DevTools → Network. After upload, watch for a request to /api/images triggered by the invalidation (SvelteKit re-runs the load function, which fetches again).
  • Hard-refresh the page — images still load (because the load function runs on navigation too).
Bonus challenge
The admin page has a delete flow (none currently, but imagine it). If you added a delete button that called DELETE /api/images/:filename, where would you call invalidate? What if you also wanted to show the count of images in a layout header? Would invalidate('/api/images') refresh that too?

When to use each pattern

Scenario Use
Submitting an HTML form, CRUD via form actions use:enhance (calls update(), which invalidates automatically)
Client fetch (file upload, JSON API call) after which page data needs refresh invalidate(url) after the fetch completes
Data from Prisma directly in a load function (no URL) depends('myapp:key') in load + invalidate('myapp:key') in component
Multiple things changed at once, not worth targeting invalidateAll()
Note on use:enhance
You're already using use:enhance in the notes page. Under the hood, when update() is called in the callback, SvelteKit runs invalidateAll() to re-run all load functions. That's why the notes list refreshes after adding a note. invalidate(url) gives you the same power but with surgical precision.
