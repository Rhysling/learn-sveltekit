1. The Adapter Model

SvelteKit is platform-agnostic. An adapter is a plugin that packages your built app for a specific deployment runtime. Official adapter docs →

Nuxt analogy
Same idea as Nuxt's Nitro presets (preset: 'node-server', preset: 'vercel'). SvelteKit just makes it an explicit import in svelte.config.ts.
adapter-node
Node.js server

Produces a self-contained Node.js HTTP server. Run with node build. Works on any VPS, container, or PaaS that runs Node — including Fly.io.

Your project uses this.

adapter-vercel
Serverless

Deploys to Vercel edge/serverless functions. No persistent filesystem — SQLite does not work here.

adapter-static
Static only

Pre-renders every page at build time. No server, no form actions, no server-side secrets. Not suitable for your app.

SQLite + serverless = no
Vercel, Netlify, and Cloudflare Workers are stateless — each request may hit a different container with a fresh filesystem. A SQLite file lives on disk. It won't be shared, and data written in one request is gone for the next. adapter-node on a long-running container (Fly machine) is the correct match for this app.

2. What npm run build Produces

$ npm run build
✓ 47 modules transformed.
build/server/index.js ← your compiled server handler
build/client/ ← browser assets (JS, CSS)
build/prerendered/ ← any static pre-rendered pages

The build/ folder is a self-contained Node.js app. Start it with:

$ node build
Listening on 0.0.0.0:3000

No Vite, no dev server, no hot reload. Plain Node. Your package.json already has "start": "node build" for exactly this. Always test this locally before deploying.

3. Environment Variables: The Critical Distinction

SvelteKit has four env modules. Picking the wrong one is the most common first-deploy mistake. Docs →

Module Resolved at Available in Use for
$env/static/private Build time (or startup with adapter-node) Server only Secrets: JWT, DB URL
$env/dynamic/private Request time Server only Secrets that vary per request / runtime
$env/static/public Build time Server + Client Public config (PUBLIC_API_URL)
$env/dynamic/public Request time Server + Client Public config resolved at runtime

The "static" naming trap

$env/static/private sounds like values are frozen at compile time — they aren't, with adapter-node. The word "static" means statically analyzable by Vite (so it can tree-shake and warn on missing vars), not that the value is baked in as a literal.

With adapter-node, $env/static/private reads from process.env when the Node server starts — exactly the same moment as secrets injected by Fly.io. Setting fly secrets set JWT_SECRET=… works correctly. adapter-node source →

Your project is already correct
src/lib/server/auth.ts and src/lib/server/db.ts import from $env/static/private. The process.env.NODE_ENV usage in cookie setup is also fine — NODE_ENV is a universal Node.js convention, not a SvelteKit secret. No env changes needed before deploying.

4. Deploying to Fly.io

Fly.io runs your app inside a lightweight VM (a "machine") anywhere in the world. It's a great fit for this project: long-running Node server, persistent filesystem volume for SQLite and uploads, no cold-start lag. The free tier includes 3 shared-CPU machines and 3 GB of volume storage. Pricing →

Install the Fly CLI (flyctl)

# Windows (PowerShell)
$ iwr https://fly.io/install.ps1 -useb | iex
# macOS / Linux
$ curl -L https://fly.io/install.sh | sh
$ fly auth login
Opening browser to authenticate…

Step-by-step deploy

  1. 1

    Test your build locally.

    $ npm run build
    $ node build
    Listening on 0.0.0.0:3000

    Visit http://localhost:3000 and test auth, notes, the image gallery. Fix any build errors before touching Fly.

  2. 2

    Make sure .env and uploads/ are in .gitignore.
    Secrets must never be committed. Uploaded files don't belong in git either.

  3. 3

    Initialize your Fly app. Run this from your project root:

    $ fly launch
    Detected a NodeJS app
    ? Choose an app name: my-sveltekit-app
    ? Choose a region: sea (Seattle) — pick one close to your users
    ? Would you like to set up a Postgresql database? No
    ? Would you like to deploy now? No ← say No, we need to configure first

    This creates fly.toml and a Dockerfile in your project.

  4. 4

    Edit the generated Dockerfile.
    fly launch generates a decent Dockerfile for Node apps, but confirm it builds and starts correctly. It should look roughly like:

    Dockerfile
    FROM node:20-slim
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --omit=dev
EXPOSE 8080
ENV PORT=8080
CMD ["node", "build"]

    If the generated file differs significantly, the key requirements are: npm run build during the image build, then node build as the start command. Prisma also needs its generated client — npm ci (which runs prisma generate via postinstall if configured) should handle it.

  5. 5

    Configure fly.toml.
    Open the generated fly.toml and make sure it includes a [[mounts]] section for the persistent volume:

    fly.toml
    [build]
  # Fly uses your Dockerfile automatically

[env]
  PORT = "8080"
  # Non-secret env vars go here
  # Secrets (JWT_SECRET, DATABASE_URL) go in fly secrets — NOT here

[[mounts]]
  source      = "data"          # name of the Fly volume
  destination = "/app/data"    # path inside the container

[[services]]
  internal_port = 8080
  protocol      = "tcp"

  [[services.ports]]
    handlers = ["http"]
    port     = 80

  [[services.ports]]
    handlers = ["tls", "http"]
    port     = 443
  6. 6

    Create the persistent volume.

    $ fly volumes create data --region sea --size 1
    ID: vol_xxxxxxxxxx
    Name: data
    Region: sea
    Size (GB): 1

    Match --region to whatever region you chose during fly launch. This 1 GB volume will hold your SQLite file and uploaded images.

  7. 7

    Set secrets.
    Secrets are encrypted at rest by Fly and injected as environment variables at runtime. They are never stored in fly.toml or your Dockerfile.

    $ fly secrets set \
    JWT_SECRET="your-strong-secret-here" \
    DATABASE_URL="file:/app/data/db.sqlite" \
    IMAGE_STORAGE_PATH="/app/data/uploads"

    Note the paths: /app/data/ is the volume mount point from fly.toml. Everything inside it survives redeploys.

  8. 8

    Deploy.

    $ fly deploy
    ==> Building image with Docker
    --> Building image done
    ==> Pushing image to fly
    ==> Creating release
    --> release v1 created
    Visit your app at https://my-sveltekit-app.fly.dev

    Fly builds the Docker image, pushes it, and starts a machine. Watch the logs live with fly logs.

  9. 9

    Run Prisma migrations.
    On first deploy — and after every schema change — apply migrations to the production database:

    $ fly ssh console -C "cd /app && npx prisma migrate deploy"
    Connecting to top1.nearest.of.my-sveltekit-app.internal... complete
    Applying migration `20240101_init`
    Done

    migrate deploy (not migrate dev) is the production command: non-interactive, applies only pending migrations, safe for CI/CD.

After a successful deploy
Future deploys are just fly deploy — one command. Fly rebuilds the Docker image, replaces the machine, and keeps the volume mounted. Your SQLite data and uploaded images are untouched.

5. The SQLite Persistence Picture

Without a volume (bad)
SQLite file lives in the container filesystem. Every fly deploy replaces the container. Your entire database is wiped with each deploy.
With a volume at /app/data (correct)
The volume is a separate disk attached to the machine. Deploying replaces the container but keeps the volume. Your data survives across all redeploys.
SQLite on Fly — one machine only
Fly volumes are attached to a single machine in a single region. If you later scale to multiple machines, SQLite won't work (two machines can't safely write to the same file). That's a future problem — for now, one machine is correct, cost-effective, and perfectly capable. If you ever need multi-machine scale, Turso is a hosted SQLite-compatible service with a Prisma driver.

6. Pre-Deploy Checklist

Check Your Understanding

1. You want to move to Vercel because their CDN is faster. Your app uses SQLite. What must change?

Correct. Vercel's filesystem is ephemeral — SQLite data disappears between requests. You'd keep Prisma but point it at a hosted Postgres or SQLite-compatible service.
Switching the adapter is necessary but not sufficient. Vercel's filesystem is ephemeral, so SQLite data vanishes. You need a hosted database too.

2. You add STRIPE_KEY via fly secrets set STRIPE_KEY=sk_live_… to a running Fly app. When does it take effect?

Correct. fly secrets set triggers a rolling restart of your machines. The new secret is available from process.env in the freshly started Node process — no manual redeploy needed.
A running Node process won't pick up new env vars mid-flight, but Fly is smarter than a raw VPS — fly secrets set automatically triggers a rolling machine restart, so the new secret is live after a brief redeploy.

3. After deploying you notice that new user registrations are failing in production but working in dev. The most likely cause is:

Correct. Fly deploys the app but doesn't touch the database. The SQLite file on the volume is empty until you run fly ssh console -C "npx prisma migrate deploy" to create the schema.
Prisma is installed via npm ci in the Dockerfile, and Prisma generates the client automatically. The most common first-deploy breakage is forgetting to run prisma migrate deploy — the database schema doesn't exist yet.
