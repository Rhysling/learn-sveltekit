Shared State & the Context API

Three patterns for sharing reactive state between components — and when to reach for each one.

&9201; ~30 min 📋 Quiz + 3 exercises 🎯 Tied to: auth state, admin features, component architecture

The Problem: State That Lives Between Components

So far, every piece of $state you've written lives inside a single component. That works for local UI — a toggle, a form field, a counter. But real apps need state that multiple components can read and write:

  • The logged-in user's name, shown in a nav bar and a dashboard page
  • A theme or locale preference, readable by every component in the tree
  • A shared list (like your admin user list) that a parent page and its child edit modals both mutate

Svelte 5 gives you three patterns for this, each suited to a different scope:

PatternScopeWhen to use
Module state
.svelte.ts files		 Global — any component can import it App-wide singletons: theme, feature flags, a shared data cache
Context API
setContext / getContext		 Subtree — parent and its descendants only Per-instance state: a form group, a data table, a layout's auth state
Props
$props()		 Direct parent → child One-hop data: passing a user object to a child card
Vs Vue / Nuxt
Vue has provide / inject (= Svelte's context), Pinia stores (= Svelte's module state in .svelte.ts), and props. The mental model maps directly — the APIs are just different.

Pattern 1: Module State in .svelte.ts Files

You already have one of these in your project: src/lib/context/dummy-store.svelte.ts. Let's break down why it works.

The .svelte.ts file extension tells the Svelte compiler to process runes in this file. Without it, $state is just an undefined variable — the compiler won't transform it. This is the key insight: runes work in any .svelte.ts / .svelte.js file, not just .svelte components.

Official $state docs →

The class pattern (recommended)

The cleanest way to share reactive state is a class with $state fields, exported as a singleton:

src/lib/state/theme.svelte.ts
class ThemeState {
  mode = $state<'light' | 'dark'>('light');

  toggle() {
    this.mode = this.mode === 'light' ? 'dark' : 'light';
  }
}

export const theme = new ThemeState();
Any component
<script>
  import { theme } from '$lib/state/theme.svelte';
</script>

<button onclick={() => theme.toggle()}>
  Current: {theme.mode}
</button>

Any component that imports theme gets the same instance. When one component calls theme.toggle(), every other component reading theme.mode re-renders automatically. No subscriptions, no .value, no ceremony.

The object pattern (simpler, same idea)

If you don't need methods, export a plain $state object:

// src/lib/state/ui.svelte.ts
export const ui = $state({
  sidebarOpen: true,
  modalVisible: false,
});

Components import it and mutate properties directly: ui.sidebarOpen = false.

You cannot export a reassignable $state primitive

This does not work:

export let count = $state(0);  // ✗ Importing modules won't see updates

The Svelte compiler transforms $state into getter/setter wrappers, but those transformations are per-file. An importer gets the raw value at import time, not the reactive wrapper. Always export an object or class instance — mutate properties, don't reassign the whole binding.

$state.raw — when you don't want deep reactivity

By default, $state wraps objects in a deep reactive proxy. If you're storing a large, read-mostly dataset (e.g. a cached API response with hundreds of rows), $state.raw skips the proxy and only reacts to reassignment:

class CachedUsers {
  users = $state.raw<User[]>([]);

  async refresh() {
    const res = await fetch('/api/users');
    this.users = await res.json();  // reassignment → triggers update
  }
}

export const cachedUsers = new CachedUsers();

cachedUsers.users.push(newUser) would not trigger an update — you'd have to reassign: cachedUsers.users = [...cachedUsers.users, newUser]. That's the tradeoff for better performance on large data.

SSR warning: module state is shared across requests

On the server, a .svelte.ts module is loaded once and shared across all incoming requests. If you mutate module-level state during SSR, one user's data could leak into another user's response. This is the same problem as a global variable in any Node server.

Rule: Module state is safe for client-only concerns (theme, UI toggles, cached data fetched client-side). For anything per-request (auth, user data), use the Context API instead — it's scoped to a component tree, not a module.

Pattern 2: The Context API

Context solves a different problem than module state. It's for state that belongs to a specific subtree of the component hierarchy — not the whole app.

+layout.svelte ← setContext('auth', user) ├─ NavBar.svelte ← getContext('auth') → shows user name ├─ +page.svelte ← getContext('auth') → uses user.id └─ Sidebar.svelte ← getContext('auth') → shows avatar

The parent calls setContext during initialization. Any descendant calls getContext with the same key. No prop drilling through intermediate components.

Official Context API docs →

The four functions

FunctionWhat it does
setContext(key, value) Stores a value in the current component's context. Descendants can retrieve it.
getContext(key) Retrieves the value set by the nearest ancestor. Throws if no ancestor set it.
hasContext(key) Returns true if an ancestor set this key. Use for optional context.
getAllContexts() Returns the full context Map. Rarely needed — useful for wrapper components that forward context.

All four must be called during component initialization — that is, synchronously in the <script> block at the top level, not inside an event handler or $effect.

Basic example: string key

Parent (sets context)
<script>
  import { setContext } from 'svelte';
  import Child from './Child.svelte';

  setContext('greeting', 'Hello!');
</script>

<Child />
Child (reads context)
<script>
  import { getContext } from 'svelte';

  const msg = getContext('greeting');
</script>

<p>{msg}</p>
<!-- renders: Hello! -->

Reactive context: passing $state objects

A plain string in context is static — the child gets a snapshot. To make context reactive, pass a $state object:

Parent layout
<script>
  import { setContext } from 'svelte';

  let counter = $state({ count: 0 });
  setContext('counter', counter);
</script>

<button onclick={() => counter.count += 1}>
  Increment ({counter.count})
</button>

<slot />
Any descendant
<script>
  import { getContext } from 'svelte';

  const counter = getContext('counter');
</script>

<p>Count is: {counter.count}</p>
<!-- Updates live when parent mutates counter.count -->
The "broken link" trap

If the parent reassigns the entire object, the child's reference breaks:

// In the parent:
counter = { count: 0 };  // ✗ child still holds the OLD object

// Instead, mutate properties:
counter.count = 0;        // ✓ child sees the update

This is the same rule as the $state destructuring gotcha from lesson 3. The child holds a reference to the reactive proxy object. Reassigning the variable in the parent creates a new proxy that the child knows nothing about.

createContext — the type-safe way (Svelte 5.40+)

String keys are fragile — typos silently break, and TypeScript can't infer the type of the value. Svelte 5.40 introduced createContext to solve both problems:

src/lib/context/auth.ts
import { createContext } from 'svelte';
import type { AuthTokenPayload } from '$lib/types/auth';

export const [getAuth, setAuth] = createContext<AuthTokenPayload>();

createContext<T>() returns a [getter, setter] tuple:

  • setAuth(value) — calls setContext internally with an auto-generated unique key
  • getAuth() — calls getContext with that same key. Throws if no ancestor called setAuth.
Parent layout
<script>
  import { setAuth } from '$lib/context/auth';

  let { data } = $props();
  setAuth(data.user);
</script>
Any descendant
<script>
  import { getAuth } from '$lib/context/auth';

  const user = getAuth();
  // ^? AuthTokenPayload — fully typed!
</script>

No string keys to mistype. Full TypeScript inference. If a child tries to call getAuth() without any ancestor having called setAuth(), it throws immediately — no silent undefined.

createContext vs setContext/getContext

createContext is the recommended approach going forward. Use raw setContext/getContext when you need dynamic keys, or when working with code that predates 5.40. For new code, prefer createContext — it's safer and cleaner.

Context is per-instance, not global

This is the key difference from module state. If you render a component twice, each instance gets its own context:

<!-- Each TabPanel sets its own context -->
<TabPanel label="Tab A">
  <TabContent />   <!-- reads Tab A's context -->
</TabPanel>

<TabPanel label="Tab B">
  <TabContent />   <!-- reads Tab B's context -->
</TabPanel>

Module state can't do this — there's only one instance. Context is the right tool whenever the "who set it" matters.

Context + SSR: why it's safe

Remember the SSR warning from module state? Context doesn't have that problem. Each server-side render creates a fresh component tree with its own context. Request A's setContext('auth', userA) is isolated from request B's. This makes context the correct choice for per-user data like auth state.

Combining Patterns: Context + Module State

In practice, you'll often use both. Here's how they'd compose in your app:

$lib/state/theme.svelte.ts ← global, client-only (module state) ├─ imported by any component that needs theme +layout.svelte (protected) ← per-request auth (context) ├─ setAuth(data.user) ├─ NavBar.svelte ← reads auth and theme └─ +page.svelte ← reads auth

The NavBar imports theme from the module (global UI concern) and calls getAuth() from context (per-request concern). Clean separation.

When To Use What — Decision Flowchart

QuestionAnswer
Is it only used by one component?Local $state — don't share it at all.
Is it one hop away (parent → child)?Props via $props().
Does it need to survive SSR without leaking across requests?Context API (setContext / createContext).
Could multiple instances of the same component have different values?Context API. Module state is a singleton.
Is it truly app-wide and client-only (theme, locale, UI toggles)?Module state in .svelte.ts.

Hands-On Exercises

These exercises build real shared-state patterns into your project. Each one is small but teaches a distinct technique.

Exercise 1 Create a notification toast with module state

Build a tiny notification system: a .svelte.ts module that any component can import to push a toast message, and a <Toast> component that displays them.

class ToastState {
  items = $state<{ id: number; message: string }[]>([]);
  nextId = 0;

  push(message: string, durationMs = 3000) {
    const id = this.nextId++;
    this.items = [...this.items, { id, message }];
    setTimeout(() => this.dismiss(id), durationMs);
  }

  dismiss(id: number) {
    this.items = this.items.filter(t => t.id !== id);
  }
}

export const toasts = new ToastState();
<script>
  import { toasts } from '$lib/state/toasts.svelte';
</script>

{#if toasts.items.length > 0}
  <div class="toast-container">
    {#each toasts.items as toast (toast.id)}
      <div class="toast">
        {toast.message}
        <button onclick={() => toasts.dismiss(toast.id)}>×</button>
      </div>
    {/each}
  </div>
{/if}
✓ Verified when: Calling toasts.push('hello') from any component shows a toast that auto-dismisses after 3 seconds. Two different pages can both trigger toasts and they all appear in the same container. This is module state in action — one global instance, any component can write to it.
Exercise 2 Share auth state via createContext

Right now your protected layout passes data.user as props or reads it from data directly. Refactor it to use the Context API so any deeply nested component can access the logged-in user without prop drilling.

import { createContext } from 'svelte';
import type { AuthTokenPayload } from '$lib/types/auth';

export const [getAuth, setAuth] = createContext<AuthTokenPayload>();
<script>
  import { setAuth } from '$lib/context/auth';

  let { data, children } = $props();
  setAuth(data.user);
</script>

{@render children()}
<script>
  import { getAuth } from '$lib/context/auth';

  const user = getAuth();
  // user is fully typed as AuthTokenPayload
</script>

<p>Logged in as: {user.name}</p>
✓ Verified when: The page still displays the correct user info. Check that TypeScript is happy (no type assertions needed). Try calling getAuth() from a component outside the (protected) group — it should throw, confirming that context is scoped to the subtree.
Exercise 3 Reactive context: a page-level edit mode

Build a small reactive context pattern: a parent page provides an "edit mode" toggle, and child components react to it without props.

import { createContext } from 'svelte';

interface EditMode {
  active: boolean;
  toggle: () => void;
}

export const [getEditMode, setEditMode] = createContext<EditMode>();
<script>
  import { setEditMode } from '$lib/context/edit-mode';

  let editMode = $state({ active: false, toggle() { this.active = !this.active; } });
  setEditMode(editMode);
</script>

<button onclick={() => editMode.toggle()}>
  {editMode.active ? 'Exit Edit Mode' : 'Enter Edit Mode'}
</button>
<script>
  import { getEditMode } from '$lib/context/edit-mode';

  const edit = getEditMode();
</script>

{#if edit.active}
  <div class="edit-controls">
    <button>Delete</button>
    <button>Rename</button>
  </div>
{/if}

<!-- Child can also toggle: -->
<button onclick={() => edit.toggle()}>
  {edit.active ? 'Done' : 'Edit'}
</button>
✓ Verified when: Clicking the toggle in either the parent or child shows/hides the edit controls in the child. Both components stay in sync because they share the same $state proxy via context. The child didn't receive any props — it pulled the state from the component tree.

What You Learned

New Concepts

  • The "can't export a reassignable $state" rule
  • $state.raw for large, read-mostly data
  • SSR module state isolation risk
  • setContext / getContext / hasContext

Decision Rules

  • Local only → local $state
  • One hop → props
  • Subtree, per-instance, SSR-safe → Context
  • App-wide, client-only → module state
  • New code → prefer createContext
  • Reactive context → pass $state object, never reassign
