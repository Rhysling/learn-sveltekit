Prisma Relations & Migrations

Connect your models with relations, evolve your schema through migrations, seed your database, and write queries that join data across tables.

Where You Are

Your project already has Prisma + SQLite working. You have a User model, a Note model, and CRUD operations that talk to the database through server load functions and form actions. But right now, your two models are isolated — there's no way to know which user created which note.

Current schema User Note ├─ id (cuid) ├─ id (autoincrement) ├─ email ├─ body ├─ password └─ createdAt ├─ name └─ ... No connection between them.

This lesson teaches you to connect them with a relation, migrate the database, seed it with test data, and write queries that pull related data in a single call.

Vs Nuxt
If you've used Prisma in a Nuxt project, the Prisma side is identical — the schema file, migration commands, and query API don't change. What's different is where you call Prisma: in SvelteKit, it's +page.server.ts load functions and form actions, not Nuxt's server/api/ routes.

Relations in Prisma

A relation connects two models so you can query across them. Prisma supports three kinds:

RelationExampleSchema syntax
One-to-many A User has many Notes Foreign key on the "many" side
One-to-one A User has one Profile Foreign key + @unique on one side
Many-to-many Posts have many Tags, Tags have many Posts Implicit join table (Prisma creates it for you)

The most common — and the one you need right now — is one-to-many. Let's add it.

Adding a one-to-many relation

To connect Note to User, you need two things in schema.prisma:

  1. A foreign key field on Note (authorId) that stores the User's id
  2. Relation fields on both sides that tell Prisma how to join them
prisma/schema.prisma
model User {
  id          String   @id @default(cuid())
  email       String   @unique
  password    String
  name        String?
  createdAt   DateTime @default(now())
  lastLoginAt DateTime @default(now())
  isAdmin     Boolean  @default(false)
  isDisabled  Boolean  @default(false)
  notes       Note[]              // ← relation field (no column in DB)
}

model Note {
  id        Int      @id @default(autoincrement())
  body      String
  createdAt DateTime @default(now())
  author    User     @relation(fields: [authorId], references: [id])  // ← relation field
  authorId  String                                                  // ← foreign key (real column)
}

There are two kinds of fields at work here:

FieldIn DB?Purpose
authorId String Yes — real column Stores the User's id. This is the foreign key.
author User @relation(...) No — virtual Tells Prisma how to join. Used by include in queries.
notes Note[] No — virtual The "back-relation" on User. Lets you query user.notes.
The @relation directive
@relation(fields: [authorId], references: [id]) means: "join Note.authorId to User.id." This is required on the side that holds the foreign key. The other side (User.notes) doesn't need it — Prisma infers the reverse.

Optional vs required relations

The schema above makes authorId required — every note must have an author. If you have existing notes without authors, the migration will fail because those rows can't satisfy the constraint.

You have two choices:

ApproachSchemaWhen to use
Optional relation author User? @relation(...)
authorId String?		 You have existing data that lacks the foreign key. Old rows keep null.
Required relation author User @relation(...)
authorId String		 Clean start, or you'll backfill/wipe existing data before migrating.

For this lesson, we'll use an optional relation since your database already has notes. You can tighten it to required later once all notes have authors.

The Migration Workflow

Right now your project has a single 0_baseline migration. That was the initial CREATE TABLE. Every schema change after that follows the same three-step cycle:

Schema change cycle 1. Edit schema.prisma → 2. prisma migrate dev → 3. prisma generate (your change) (creates SQL + runs it) (updates TypeScript types)

Step by step

1 Edit the schema. Add the relation fields to schema.prisma as shown above.

2 Create and run the migration.

npx prisma migrate dev --name add-note-author

This command does three things:

  1. Diffs your schema against the database and generates a SQL migration file
  2. Runs the SQL against your dev database
  3. Regenerates the Prisma client (so you get updated TypeScript types)

The migration file lands in prisma/migrations/<timestamp>_add_note_author/migration.sql. It's committed to git — other developers (or your production deploy) run prisma migrate deploy to apply it.

3 Verify. Check the generated SQL file to make sure it did what you expected:

-- Expected: adds a nullable column, not a destructive change
ALTER TABLE "Note" ADD COLUMN "authorId" TEXT;

-- Adds a foreign key constraint
-- (SQLite handles this via PRAGMA foreign_keys)
SQLite migration quirk
SQLite doesn't support ALTER TABLE ... ADD CONSTRAINT. When Prisma needs to add a foreign key to an existing table in SQLite, it may recreate the table — create a new table with the constraint, copy data, drop the old one, rename. This is safe but can be surprising when you read the migration SQL. PostgreSQL/MySQL don't have this limitation.

What if you mess up?

In development, you have escape hatches:

SituationCommand
Want to start over completely npx prisma migrate reset — drops the database, re-runs all migrations, then seeds
Schema is out of sync, no data to save npx prisma db push — pushes schema directly without creating a migration file (prototyping only)
Need to see what SQL would be generated npx prisma migrate diff --from-migrations prisma/migrations --to-schema-datamodel prisma/schema.prisma
Never use db push in production
prisma db push skips migration history. It's fine for prototyping, but production must use prisma migrate deploy so changes are tracked, reviewable, and reproducible.

Seeding the Database

Your prisma.config.ts already references a seed script (tsx prisma/seed.ts), but the file doesn't exist yet. A seed script populates your dev database with realistic test data so you're not starting from an empty slate after every migrate reset.

prisma/seed.ts
import { PrismaClient } from '../src/generated/prisma/client';
import { PrismaBetterSqlite3 } from '@prisma/adapter-better-sqlite3';
import { hashSync } from 'bcryptjs';

const adapter = new PrismaBetterSqlite3({
  url: process.env.DATABASE_URL ?? 'file:./dev.db',
});
const prisma = new PrismaClient({ adapter });

async function main() {
  // Create users
  const alice = await prisma.user.upsert({
    where: { email: 'alice@example.com' },
    update: {},
    create: {
      email: 'alice@example.com',
      name: 'Alice',
      password: hashSync('password123', 10),
      isAdmin: true,
    },
  });

  const bob = await prisma.user.upsert({
    where: { email: 'bob@example.com' },
    update: {},
    create: {
      email: 'bob@example.com',
      name: 'Bob',
      password: hashSync('password123', 10),
    },
  });

  // Create notes with authors
  await prisma.note.createMany({
    data: [
      { body: 'First note from Alice', authorId: alice.id },
      { body: 'Second note from Alice', authorId: alice.id },
      { body: 'Bob\'s first note', authorId: bob.id },
    ],
  });

  console.log('Seeded: 2 users, 3 notes');
}

main()
  .catch((e) => { console.error(e); process.exit(1); })
  .finally(() => prisma.$disconnect());
Why upsert?
upsert means "create if it doesn't exist, otherwise update." This makes the seed script idempotent — you can run it multiple times without creating duplicate users. The notes use createMany, which will add duplicates on re-run; if that bothers you, delete all notes first with prisma.note.deleteMany().

The seed runs automatically after prisma migrate reset, or manually with:

npx prisma db seed

Querying Related Data

With the relation in place, Prisma gives you two ways to pull related data: include and select.

include — load the whole related object

+page.server.ts
const notes = await prisma.note.findMany({
  include: { author: true },     // ← joins User table
  orderBy: { createdAt: 'desc' },
});

// Each note now has note.author.name, note.author.email, etc.

include: { author: true } tells Prisma to join the User table and attach the full user object to each note. The generated SQL is a LEFT JOIN (or a second query, depending on the adapter).

select — pick specific fields

If you don't need the whole User object, select lets you cherry-pick fields. This is more efficient and avoids leaking sensitive data (like password):

const notes = await prisma.note.findMany({
  select: {
    id: true,
    body: true,
    createdAt: true,
    author: {
      select: { name: true, email: true },  // ← only name and email
    },
  },
  orderBy: { createdAt: 'desc' },
});
include vs select: you can't use both at the top level
Prisma doesn't allow include and select on the same query level. If you use select, you must explicitly list every field you want — nothing is included by default. If you use include, you get all scalar fields plus the included relations.

The reverse direction: querying a user's notes

const userWithNotes = await prisma.user.findUnique({
  where: { id: userId },
  include: { notes: true },  // ← loads all notes by this user
});

// userWithNotes.notes is Note[]

This works because of the notes Note[] back-relation field on User. Prisma handles the join in both directions.

Nested writes: creating a note with its author in one call

Instead of manually setting authorId, you can use Prisma's nested write syntax to connect related records:

Manual (set the FK)
await prisma.note.create({
  data: {
    body: 'Hello',
    authorId: userId,
  },
});
Nested write (connect)
await prisma.note.create({
  data: {
    body: 'Hello',
    author: {
      connect: { id: userId },
    },
  },
});

Both do the same thing. The manual version is simpler for one-to-many. connect shines when creating deeply nested structures, or when you want to create a related record at the same time:

// Create a user AND their first note in a single transaction
await prisma.user.create({
  data: {
    email: 'carol@example.com',
    password: hashedPw,
    name: 'Carol',
    notes: {
      create: [
        { body: 'Carol\'s first note' },
        { body: 'Carol\'s second note' },
      ],
    },
  },
});

Why Your db.ts Uses a Global Singleton

You already have this pattern in src/lib/server/db.ts:

const prisma = global.prisma ?? new PrismaClient({ adapter });
if (process.env.NODE_ENV !== 'production') global.prisma = prisma;

This exists because of Vite's HMR (hot module replacement). During development, when you save a file, Vite re-imports changed modules. Without the global cache, each hot reload would create a new PrismaClient instance, eventually exhausting database connections.

The global singleton pattern stores the client on globalThis, which survives HMR. In production, there's no HMR, so you get a single client naturally.

Vs Nuxt
Same problem, same pattern. Nuxt's HMR has the same module re-import behavior. If you've used Prisma in a Nuxt server/utils/db.ts, you've likely seen the same globalThis.prisma trick.

SvelteKit Integration Patterns

Here's how relations fit into the SvelteKit patterns you already know:

Load function → include related data

src/routes/examples/notes/+page.server.ts
export const load: PageServerLoad = async ({ locals }) => {
  return {
    notes: await prisma.note.findMany({
      include: { author: { select: { name: true } } },  // safe: no password leak
      orderBy: { createdAt: 'desc' },
    }),
  };
};
Never include password in load returns
SvelteKit serializes load function return values and sends them to the client. If you include: { author: true }, the full User object — including the hashed password — ends up in the HTML payload. Always use select inside include to pick only safe fields.

Form action → create with relation

Form action: add a note for the logged-in user
add: async ({ request, locals }) => {
  if (!locals.user) return fail(401, { error: 'Not logged in' });

  const data = await request.formData();
  const body = String(data.get('body') ?? '').trim();

  if (!body) return fail(400, { error: 'Note body is required.' });

  await prisma.note.create({
    data: { body, authorId: locals.user.userId },
  });
},

The locals.user object is set by your auth hook (hooks.server.ts), which runs before every request. This is the standard pattern: the hook verifies the JWT and attaches the user; the load function or action uses locals.user to scope queries.

Quiz — Check Your Understanding

Click an option to check immediately.

1 You add author User @relation(fields: [authorId], references: [id]) to the Note model. Which field actually exists as a column in the database?
A author — it's a User reference column.
B authorId — the relation field is virtual; only the foreign key is a real column.
C Both — Prisma creates columns for both fields.
The author User @relation(...) field is virtual — it tells Prisma how to join tables in queries, but no column is created for it. Only authorId exists in the database as a real column storing the foreign key value.
Correct. Relation fields like author and notes are virtual — they exist only in the Prisma schema to describe relationships. The authorId String field is the real database column.
2 You run prisma.note.findMany({ include: { author: true } }) and return the result from a server load function. What's the security risk?
A No risk — server load functions keep data server-side.
B The full User object (including hashed password) gets serialized into the HTML payload sent to the client.
C It causes a SQL injection vulnerability.
SvelteKit serializes the return value of server load functions and embeds it in the HTML sent to the browser. include: { author: true } pulls the full User object — including password. Use select inside include to pick only safe fields.
Correct. SvelteKit's server load functions serialize their return value into the client-side HTML. include: { author: true } brings the full User, password hash included. Always use include: { author: { select: { name: true } } } to pick safe fields.
3 You want to add a required authorId column to the Note table, but you have existing notes in the database. What happens when you run prisma migrate dev?
A The migration fails because existing rows have no value for the new required column.
B Prisma auto-fills existing rows with a default value.
C It silently deletes existing notes without an author.
A required column with no default cannot be added to a table that already has rows — those rows would violate the NOT NULL constraint. The migration will fail. Use an optional relation (String?) to allow nulls for existing data, or wipe the data first.
Correct. Existing rows can't satisfy a NOT NULL constraint for a column they don't have. Either make the relation optional (authorId String?), add a @default, or clear existing data before migrating.
4 What does upsert do in a seed script?
A Deletes the record if it exists, then creates a new one.
B Creates the record if it doesn't exist; if it does, applies the update clause (which can be empty).
C Inserts multiple records at once, skipping duplicates silently.
upsert = "update or insert." It checks the where condition. If a match exists, it applies the update clause. If no match, it runs the create clause. With update: {}, it's effectively "create if missing, otherwise leave it alone" — making the seed idempotent.
Correct. upsert checks where, then either updates or creates. With update: {}, it's a safe "create if not exists" — perfect for seeds that can run multiple times without creating duplicates.
5 Why does db.ts store the Prisma client on globalThis in development?
A To make Prisma available in client-side code.
B To prevent Vite's HMR from creating a new PrismaClient on every file save, which would exhaust database connections.
C To share the Prisma instance across multiple worker threads.
The global singleton exists because Vite's hot module replacement re-imports modules on every file save. Without caching on globalThis, each reload creates a new PrismaClient, and you quickly run out of database connections. In production there's no HMR, so this isn't needed.
Correct. Vite's HMR re-imports modules on save. Without the global cache, each hot reload spawns a new PrismaClient. globalThis survives HMR, so the same client is reused across reloads.
6 Can you use include and select together at the top level of a Prisma query?
A No — they're mutually exclusive at the same level. Use select inside include (or vice versa) for nested relations.
B Yes — include adds relations and select picks scalars, so they complement each other.
C Only in Prisma 7+.
Prisma doesn't allow both include and select at the same query level. With include, you get all scalars plus specified relations. With select, you only get what you list. You can nest them: include: { author: { select: { name: true } } }.
Correct. At any given level, it's one or the other. The pattern include: { author: { select: { name: true } } } is the way to include a relation but pick specific fields from it.

Key Rules
