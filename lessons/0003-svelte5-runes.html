Lesson 3 · Svelte 5

$state, $derived, $effect

The Svelte 5 reactivity model — and how to think in runes instead of Vue composables.

⏱ ~25 min 📋 Quiz + 3 exercises 🎯 Tied to: idiomatic Svelte, dynamic UI

You already use $props() and have touched $state once (the isSubmitting flag on the notes page). This lesson completes the picture: the three runes you'll reach for constantly when building reactive components.

$state
≈ Vue ref() / reactive()

Declares reactive local state. Mutating it updates the UI.

$derived
≈ Vue computed()

A value that automatically recalculates when its dependencies change.

$effect
≈ Vue watchEffect()

Runs code as a side effect when reactive values change.

$state — Reactive Variables

The Svelte 4 way was let count = 0 — any top-level let in a <script> was implicitly reactive. Svelte 5 makes it explicit with a rune:

Svelte 4
// implicit reactivity — any let
let count = 0;
let name = 'Alice';
Svelte 5
// explicit — only rune vars are reactive
let count = $state(0);
let name  = $state('Alice');

The $state call is a compiler hint — not a runtime function. It tells Svelte to track changes to this variable and update the DOM when it changes. You still mutate it like a normal variable:

let count = $state(0);
count += 1;  // triggers UI update — no .value needed (unlike Vue ref)
Vs Vue
In Vue you'd write const count = ref(0) and read it as count.value. In Svelte 5, $state needs no .value — you use the variable directly. This is one of Svelte's key ergonomic wins.

Deep reactivity with objects and arrays

When you pass an object or array to $state, Svelte wraps it in a reactive proxy — mutations to nested properties trigger updates automatically:

let user = $state({ name: 'Alice', age: 30 });
let tags = $state(['svelte', 'typescript']);

user.age += 1;          // ✓ UI updates
tags.push('sveltekit'); // ✓ UI updates — array methods work
The destructuring gotcha
If you destructure a $state object, the extracted values are snapshots — they break reactivity: 
let user = $state({ name: 'Alice' });
const { name } = user;  // ✗ name is now a plain string — won't update
user.name = 'Bob';         // name variable still reads 'Alice'
Always read through the reactive object: user.name.

$derived — Computed Values

If a value depends on state and should stay in sync automatically, use $derived — not $state. Svelte tracks what state you read inside the expression and re-runs it whenever that state changes.

Svelte 4 reactive statement
let count = 0;
$: doubled = count * 2;
Svelte 5
let count   = $state(0);
let doubled = $derived(count * 2);

For anything beyond a single expression, use $derived.by which takes a function:

let items = $state([3, 1, 4, 1, 5]);

let sorted = $derived.by(() => {
  return [...items].sort((a, b) => a - b);
});
Vs Vue
This is Vue's computed() — same mental model, cleaner syntax. The key difference: $derived expressions must be pure (no side effects, no state mutations). If you need side effects, that's $effect.
$derived is lazy
Svelte uses a "push-pull" model: when a dependency changes, $derived is marked dirty immediately (push), but the expression only re-runs when the value is actually read (pull). This means unused derived values don't waste computation.

$effect — Side Effects

$effect is for code that needs to run in response to state changes but isn't computing a new value — things like syncing with the DOM, calling a third-party library, logging, or updating the browser title.

let query = $state('');

$effect(() => {
  // runs after mount, then again whenever `query` changes
  console.log('Search query changed:', query);
});

Cleanup — returning a teardown function

If your effect creates a subscription, interval, or event listener, return a cleanup function. Svelte calls it before the next run and when the component unmounts:

let ms = $state(1000);

$effect(() => {
  const id = setInterval(() => tick(), ms);
  return () => clearInterval(id);  // cleanup runs when ms changes or component unmounts
});

Three rules for $effect

  1. Browser-only. Effects never run during server-side rendering. Don't use them for initial data — use load functions for that.
  2. Synchronous tracking only. Svelte only tracks state read synchronously in the effect body. State read after an await is not tracked.
  3. Runs after DOM updates. The effect fires after the DOM has been updated — so document.querySelector will reflect the new state.
Vs Vue
This is Vue's watchEffect() — auto-tracks dependencies synchronously. In Vue you use onCleanup(fn) for cleanup; in Svelte 5 you return a function instead. Both run immediately after mount.

The Most Important Rule: Don't Use $effect to Derive Values

This is the single most common mistake coming from Vue. In Vue, it's tempting to use a watcher to sync values. In Svelte 5, that's an anti-pattern:

✗ Anti-pattern
let text    = $state('');
let charCount = $state(0);

$effect(() => {
  charCount = text.length; // ✗
});
✓ Idiomatic Svelte 5
let text      = $state('');
let charCount = $derived(text.length);

// charCount stays in sync automatically
// no effect needed

Reserve $effect for things that genuinely can't be expressed as a derived value: DOM manipulation, third-party library calls, browser APIs.

Which Rune Should I Use?

SituationUse
I need a variable the user can change (input, counter, toggle)$state
I need a value that depends on other state (count, total, filtered list)$derived
I need to run code when state changes (update document.title, call an API, sync to a library)$effect
I need data from the serverload function — not runes
I need to pass data from parent to child$props

Click an option to check immediately.

1 You write let todos = $state([{ done: false, text: 'buy milk' }]), then const first = todos[0]. You later run todos[0].done = true. Does first.done update?
A No — destructuring extracts a snapshot; first is no longer reactive
B Yes — $state proxies are shared by reference
C It depends on whether you use const or let
D Yes — arrays always share references in JavaScript
Close, but actually array element access (todos[0]) does return the reactive proxy object, so first.done would reflect changes since you're still reading through the proxy. The destructuring gotcha is more pronounced when you do const { done } = todos[0] — that extracted primitive breaks reactivity.
Correct — and worth understanding precisely: const first = todos[0] still holds the proxy object, so first.done would update. But const { done } = todos[0] extracts a plain boolean — that breaks reactivity. The danger is primitive extraction, not object reference.
2 You need a filtered list that updates whenever items or searchQuery change. Which is the correct approach?
A Use $state for filtered and update it in a $effect
B Use $derived.by(() => items.filter(...))
C Call a function directly in the template: {getFiltered(items, query)}
D Use $state and manually update it on every mutation
Option A is the anti-pattern — using $effect to sync derived state. Option C works but recomputes on every render cycle. $derived.by is the idiomatic answer: it caches the result and only recomputes when items or searchQuery actually change.
Correct! $derived.by is perfect here — multi-statement logic, auto-tracks both items and searchQuery, caches the result, and never requires an effect.
3 Does $effect run during server-side rendering (SSR)?
A Yes — it runs on both server and client
B No — $effect is browser-only
C Only if you use $effect.pre
D Yes, but only during prerendering
Effects are browser-only. The server renders HTML and has no DOM to interact with, so Svelte skips all $effect calls during SSR. If you need server-side initialization, use a load function.
Correct. $effect never runs on the server. This is why you shouldn't use effects for data fetching — use +page.server.ts load functions instead, which run on the server where they belong.
4 Your $effect sets up an event listener. How do you clean it up when the component is destroyed?
A Use the onDestroy lifecycle function from svelte
B Pass a second argument to $effect as the cleanup
C Return a cleanup function from the $effect callback
D Svelte handles cleanup automatically — nothing needed
While onDestroy still works in Svelte 5 for backwards compat, the idiomatic way is to return a cleanup function from the effect itself. It's called before the next effect run and when the component unmounts. No second argument exists.
Correct! Return a teardown function: $effect(() => { const el = ...; return () => el.removeEventListener(...); }). Svelte calls it before the next effect run and on unmount. This co-locates setup and teardown.
5 You have let name = $state(''). Inside a $effect, you do: const upper = name.toUpperCase(); await someApi(upper);. Is upper tracked as a dependency?
A Yes — name is read synchronously before the await, so it's tracked
B No — Svelte only tracks values read after the first await
C No — upper is a derived string, not a $state variable
D It depends on whether the effect is async
Svelte tracks reactive values read synchronously in the effect. Since name is read on the line before await, it is synchronously read and therefore tracked. The tracking window closes at the first await.
Correct. name is read synchronously, so it's tracked. Any reactive value read after an await would not be tracked — that's the synchronous-only rule in action.
6 In Vue you'd write const len = computed(() => text.value.length). What's the Svelte 5 equivalent (assuming text = $state(''))?
A let len = $state(text.length)
B $effect(() => { len = text.length })
C let len = $derived(text.length)
D let len = $derived(() => text.length)
Option A captures the value at declaration time and won't update. Option B is the anti-pattern. Option D passes a function — but $derived takes an expression directly; the function form is $derived.by(() => ...). The correct answer is C.
Correct! No .value, no function wrapper — just $derived(expression). Svelte reads the expression, tracks whatever state is accessed, and re-evaluates automatically. Cleaner than Vue's computed().

Hands-On Exercises

All three exercises are in your notes page: src/routes/examples/notes/+page.svelte. By the end, the textarea will have a live character counter, a limit indicator, and a dynamic page title.

Exercise 1 Bind the textarea to $state and derive a character count

Right now the textarea uses a one-way value={form?.body ?? ''}. Switch it to two-way binding via $state, then derive the count.

let body      = $state(form?.body ?? '');
let charCount  = $derived(body.length);
const LIMIT    = 200;
let isOverLimit = $derived(charCount > LIMIT);
<p class="char-count" class:over={isOverLimit}>
  {charCount} / {LIMIT}
</p>
✓ Verified when: Typing in the textarea shows a live "X / 200" counter. Typing past 200 chars turns the counter red and disables the submit button. Notice that charCount and isOverLimit update with zero extra code — $derived handles it.
Exercise 2 Use $effect to update the browser tab title

A small but visible $effect: update document.title as the user types, and restore it when they clear the field.

$effect(() => {
  document.title = body.length > 0
    ? `Writing... (${charCount} chars)`
    : 'Notes';
  return () => { document.title = 'Notes'; };
});
✓ Verified when: Watch the browser tab as you type in the textarea — it updates live. Clear the textarea and the title resets to "Notes". This is a genuine side effect (touching the DOM outside Svelte's control) — exactly the right use case for $effect.
Exercise 3 Spot the anti-pattern and fix it

This exercise trains the reflex. Read the bad code, understand why it's wrong, then write the fix.

let items    = $state(['apple', 'banana', 'cherry']);
let filtered = $state(items);
let query    = $state('');

$effect(() => {
  filtered = items.filter(i => i.includes(query));
});
let items    = $state(['apple', 'banana', 'cherry']);
let query    = $state('');
let filtered = $derived.by(() => items.filter(i => i.includes(query)));

// No $effect needed. filtered updates automatically.
✓ The rule it breaks: $effect is being used to synchronize a derived value. filtered is fully calculable from items and query — it has no side effects and needs no state of its own. $derived.by eliminates the effect entirely, removes an extra $state, and is more efficient (lazy, cached, no extra renders).
