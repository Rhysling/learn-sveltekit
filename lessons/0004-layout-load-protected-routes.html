Layout Load Functions &
Protected Routes

DRY auth guards using +layout.server.ts and route groups — one redirect to rule them all.

The problem with per-page auth checks

You already protect /admin by checking locals.user in its +page.server.ts:

// src/routes/admin/+page.server.ts — works, but doesn't scale
export const load: PageServerLoad = async ({ locals }) => {
  if (!locals.user) throw redirect(303, '/');
  // ... rest of load
}

This is fine for one page. Now imagine adding /admin/users, /admin/settings, /admin/reports. You'd copy that if (!locals.user) check into every single load function — a classic DRY violation.

The fix: hoist the auth check to a layout that owns the entire protected section.

Route groups — cosmetic directories

SvelteKit lets you wrap any set of routes in a parenthesised directory like (protected). The parentheses tell SvelteKit: this folder exists for organisation only — don't include its name in the URL.

src/routes/
├── (protected)/ ← parentheses = invisible to URL
│ ├── +layout.server.ts ← auth guard lives here
│ ├── dashboard/
│ │ └── +page.svelte ← URL: /dashboard
│ └── settings/
│ └── +page.svelte ← URL: /settings
├── admin/ ← URL: /admin (unprotected folder)
│ └── +page.server.ts
└── +page.svelte ← URL: /

The URL for (protected)/dashboard/+page.svelte is simply /dashboard — the route group name is invisible. You can have multiple route groups at the same level ((protected), (public), (marketing)) to apply different layouts or guards to different sections.

Nuxt analogy

In Nuxt you'd write a route middleware in middleware/auth.ts and then apply it via definePageMeta({ middleware: 'auth' }) on every protected page. In SvelteKit, +layout.server.ts in a route group is the middleware and its application in one file — no per-page annotation needed.

How +layout.server.ts works

Every +layout.server.ts exports a load function. SvelteKit runs it before any child page's load function — and if it throws a redirect(), the child never runs at all.

// src/routes/(protected)/+layout.server.ts
import { redirect } from '@sveltejs/kit';
import type { LayoutServerLoad } from './$types';

export const load: LayoutServerLoad = async ({ locals }) => {
  if (!locals.user) {
    redirect(302, '/login');  // halts everything, child never loads
  }

  return {
    user: locals.user   // available in every child page via $page.data
  };
};
throw vs return for redirect

In older SvelteKit you wrote throw redirect(...). In recent versions redirect() is a thrown error internally, so both throw redirect() and just redirect() work. The convention is to omit throw in modern code — but either compiles fine. Your existing /admin code uses throw redirect(303, '/'), which is perfectly valid.

Data flows down — always

Whatever a layout's load returns gets merged into $page.data for every child page in its scope. You don't have to do anything special in the child — the data is just there.

// src/routes/(protected)/dashboard/+page.svelte
<script lang="ts">
  import type { PageProps } from './$types';
  let { data }: PageProps = $props();
  // data.user comes from the layout load — no extra fetch needed
</script>

<h1>Welcome, {data.user.name}</h1>

If the page also has a +page.server.ts that returns its own data (say, posts), both are merged: data will have both user (from layout) and posts (from page). No conflict unless you use the same key.

The full execution order

For a request to /dashboard inside (protected), SvelteKit runs:

  1. hooks.server.ts → sets locals.user
  2. (protected)/+layout.server.ts → checks locals.user, redirects or returns { user }
  3. dashboard/+page.server.ts → loads page-specific data
  4. Both data objects merged → available in +page.svelte as data

The layout component (optional)

A route group can also have a +layout.svelte to wrap all child pages in shared UI — a sidebar, a nav bar, an admin chrome. You don't have to — a +layout.server.ts without a matching +layout.svelte is perfectly valid, just running server logic.

<!-- src/routes/(protected)/+layout.svelte -->
<script lang="ts">
  import type { LayoutProps } from './$types';
  let { data, children }: LayoutProps = $props();
</script>

<nav>Logged in as {data.user.name}</nav>
{@render children()}  <!-- renders the child page -->

{@render children()} is the Svelte 5 way to render child content — equivalent to <slot /> in Svelte 4 / Vue.

Hands-on exercises

Three steps. Apply the pattern in your real project.

1
Create the route group and layout load

Create the directory src/routes/(protected)/ and add +layout.server.ts inside it:

// src/routes/(protected)/+layout.server.ts
import { redirect } from '@sveltejs/kit';
import type { LayoutServerLoad } from './$types';

export const load: LayoutServerLoad = async ({ locals }) => {
  if (!locals.user) {
    redirect(302, '/');
  }
  return { user: locals.user };
};

This single file now guards every route you put inside (protected)/.

2
Create a protected dashboard page

Create src/routes/(protected)/dashboard/+page.svelte. This page's URL will be /dashboard:

<!-- src/routes/(protected)/dashboard/+page.svelte -->
<script lang="ts">
  import type { PageProps } from './$types';
  let { data }: PageProps = $props();
</script>

<h1>Dashboard</h1>
<p>Welcome back, {data.user.name}!</p>
<p>Logged in as: {data.user.email}</p>

Notice: no auth check in this file. The layout handles it. The data.user is available because the layout load returned it.

3
Test both logged-in and logged-out states

Logged out: Visit http://localhost:5173/dashboard without logging in. You should be redirected to /.

Logged in: Log in via your existing auth flow, then visit /dashboard. You should see your name and email rendered by the page.

Check your terminal for any server errors if something unexpected happens — remember, console.log in server files only appears there, not in the browser.

Bonus: Add a second page at (protected)/profile/+page.svelte that also shows data.user. Confirm it's also protected — no extra auth code needed in that file.

What you now know

  • Route groups ((name)/) organise routes without changing URLs
  • +layout.server.ts runs before child pages — it can redirect, and its return value is merged into child data
  • One redirect in a layout = every route in that group is protected — no repetition
  • Data inheritance is automatic — layout data is always available in child data
  • The execution chain: hooks → layout load → page load
Questions? Ask your teacher. Anything unclear — why redirect() halts the chain, how multiple nested layouts stack, how to type data.user without redeclaring it in every page — ask in the chat. That's what the teacher is here for.
