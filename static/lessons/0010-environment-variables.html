Environment Variables & Config

SvelteKit's four $env modules, when to use each, and how to stop using raw process.env in your project.

⏱ ~25 min 📋 Quiz + 3 exercises 🎯 Tied to: secrets handling, deployment safety, server/client boundary

Where You Are

Your project already uses environment variables — DATABASE_URL, JWT_SECRET, and IMAGE_STORAGE_PATH — but in two inconsistent ways:

Current state of your project db.ts → import { env } from '$env/dynamic/private' ✔ auth.ts → import { env } from '$env/dynamic/private' ✔ storage.ts → process.env.IMAGE_STORAGE_PATH ✘ login/ → process.env.NODE_ENV ✘ register/ → process.env.NODE_ENV ✘

This lesson teaches you why the $env modules exist, how they differ, and gives you a framework for choosing the right one every time. By the end, you'll fix the inconsistencies above.

Vs Nuxt
Nuxt has runtimeConfig (server) and publicRuntimeConfig (client), configured in nuxt.config.ts and accessed via useRuntimeConfig(). SvelteKit skips the config object entirely — you import directly from typed modules. The mental model is the same (private vs public, build-time vs runtime), but the mechanism is different.

Why Not Just Use process.env?

You can use process.env in server-side SvelteKit code and it works. But it has real downsides:

  1. No type safety. process.env.JWT_SECRET is string | undefined — TypeScript can't know if the variable exists. SvelteKit's $env modules generate types from your .env file, so you get autocompletion and compile-time errors.
  2. No server/client boundary enforcement. Nothing stops you from accidentally importing a file that reads process.env into a .svelte component. SvelteKit's private modules throw a build error if you try to import them client-side.
  3. No adapter portability. process.env is a Node.js API. If you ever deploy to Cloudflare Workers, Deno, or Vercel Edge, it doesn't exist. The $env modules abstract this away — they work across all adapters. (Docs: $env/dynamic/private)
The one exception
process.env.NODE_ENV is special. Vite replaces it at build time in all contexts, so it works everywhere. But it's the only process.env value you can rely on universally. For everything else, use $env.

The Four $env Modules

SvelteKit gives you four modules, organized on two axes: when the value is read (static vs dynamic) and who can see it (private vs public).

Private (server-only)Public (server + client)
Static
Read at build time		 $env/static/private $env/static/public
Dynamic
Read at runtime		 $env/dynamic/private $env/dynamic/public

Static vs Dynamic

Static modules are resolved when you run vite build. The values are baked into the JavaScript bundle. This enables dead code elimination — if you have if (env.FEATURE_FLAG === 'on'), the build can strip the else branch entirely.

Dynamic modules are read at runtime, every time the server starts. This means you can change an env var and restart the server without rebuilding. With adapter-node (which your project uses), $env/dynamic/private is equivalent to process.env. (Docs: $env/dynamic/private)

Static — baked in at build
// Values frozen when you run `vite build`
import { API_KEY } from '$env/static/private';

// Named exports — tree-shakeable
console.log(API_KEY);
Dynamic — read at server start
// Values read from process.env at runtime
import { env } from '$env/dynamic/private';

// Object — access by key
console.log(env.API_KEY);

Notice the import style difference: static modules give you named exports (one per variable), while dynamic modules give you an env object you index into.

Private vs Public

Private modules can only be imported in server-side files (+page.server.ts, +server.ts, hooks.server.ts, anything under $lib/server/). If you try to import them in a .svelte component or a universal load function, SvelteKit throws a build error. This is your safety net against leaking secrets.

Public modules are accessible everywhere — server and client. Only variables prefixed with PUBLIC_ (by default) are exposed through these modules. This prefix acts as a deliberate opt-in: you're saying "yes, I intend for the browser to see this."

Secret leak protection
If you put JWT_SECRET in your .env and try to import it from $env/static/public, it won't be there — it doesn't have the PUBLIC_ prefix. And if you try to import $env/static/private in a .svelte file, the build fails. Both paths are blocked. This is much safer than process.env, which has no guardrails.

How .env Files Work

SvelteKit uses Vite's .env loading. During development (vite dev) and at build time (vite build), Vite reads these files from your project root:

FileWhen loadedGit?
.env Always Usually no — contains real secrets
.env.local Always, overrides .env No
.env.development Only during vite dev Optional
.env.production Only during vite build Optional
.env.example Never loaded — documentation only Yes — shows team what vars are needed

Your project has both .env and .env.example. The .env has your actual dev values; .env.example is committed so collaborators know which variables to set.

Important: .env in production
In production with adapter-node, the built server reads from the real environment (process.env), not from .env files. The .env file is only used during development and build. In production, you set env vars via your hosting platform (e.g. fly secrets set, Vercel dashboard, Docker -e flags). You covered this in Lesson 7.

Type Generation

When you run vite dev or vite build, SvelteKit reads your .env file and generates TypeScript types for the $env modules. These live in .svelte-kit/types/.

This means if your .env contains DATABASE_URL and JWT_SECRET, you get autocomplete when importing from $env/static/private:

// TypeScript knows these exist because they're in .env
import { DATABASE_URL, JWT_SECRET } from '$env/static/private';
//       ^ autocomplete works here

// This would be a TypeScript error:
import { TYPO_VAR } from '$env/static/private';
//       ^ Property 'TYPO_VAR' does not exist
Declare vars even if empty
If a variable isn't in your .env during development, it won't appear in the generated types — even if it will exist in production. Add it to your .env with an empty value (STRIPE_KEY="") so the types are generated. You can override with real values via .env.local or command line: STRIPE_KEY="sk_test_..." npm run dev. (Docs)

Which Module Do I Use?

Two questions get you to the right module every time:

Decision flowchart Q1: Should the browser see this value? ├─ Yes → it needs the PUBLIC_ prefix → public module └─ No → no prefix needed → private module Q2: Can the value change between deploys without rebuilding? ├─ Yes → dynamic (read at runtime) └─ No → static (baked into bundle)

In practice, for apps deployed with adapter-node:

VariableModuleWhy
DATABASE_URL $env/dynamic/private Secret, may differ per environment without rebuild
JWT_SECRET $env/dynamic/private Secret, must be rotatable without rebuild
IMAGE_STORAGE_PATH $env/dynamic/private Server path, varies by environment
PUBLIC_APP_NAME $env/static/public Display string, safe for browser, rarely changes
PUBLIC_API_BASE $env/dynamic/public Safe for browser, but may differ between staging/prod
Rule of thumb
When in doubt, start with $env/dynamic/private. It's the safest default: server-only, runtime-read. You can always move to static later if you need tree-shaking, or to public if you need client access. Going the other direction (accidentally exposing a secret) is the mistake you can't undo.

Config: Controlling Prefixes

The PUBLIC_ prefix and other env behavior is configured in svelte.config.ts under kit.env:

// svelte.config.ts
const config = {
  kit: {
    env: {
      dir: '.',          // where to find .env files (default: project root)
      publicPrefix: 'PUBLIC_', // default — you rarely change this
      privatePrefix: '',       // default — empty means "everything not public"
    }
  }
};

With the default privatePrefix: '', every env var without the PUBLIC_ prefix is available in private modules. If you set privatePrefix: 'SECRET_', then only vars starting with SECRET_ would appear — and vars matching neither prefix would be discarded entirely. Most projects leave the defaults alone. (Docs: Configuration)

Your Project: What to Fix

Your codebase has three files still using raw process.env where they should use $env:

1. src/lib/server/storage.ts

Current (process.env)
const storagePath =
  process.env.IMAGE_STORAGE_PATH
  ?? (process.env.NODE_ENV === 'production'
    ? '/data/uploads'
    : './uploads');
Fixed ($env)
import { env } from '$env/dynamic/private';
import { dev } from '$app/environment';

const storagePath =
  env.IMAGE_STORAGE_PATH
  ?? (dev ? './uploads'
        : '/data/uploads');

Note the bonus: $app/environment exports dev and building booleans. Use dev instead of checking NODE_ENV directly — it works on all adapters, not just Node.

2. src/routes/api/login/+server.ts and register/+server.ts

Both files use process.env.NODE_ENV === 'production' for the cookie secure flag:

Current
cookies.set('token', token, {
  httpOnly: true,
  path: '/',
  secure: process.env.NODE_ENV === 'production',
  maxAge: 60 * 60 * 24 * 7
});
Fixed
import { dev } from '$app/environment';

cookies.set('token', token, {
  httpOnly: true,
  path: '/',
  secure: !dev,
  maxAge: 60 * 60 * 24 * 7
});
$app/environment
This SvelteKit module exports three values: dev (true during vite dev), building (true during vite build), and version (from your config). It works on both server and client, on all adapters. Prefer it over NODE_ENV checks. (Docs: $app/environment)

Public Variables in Practice

You haven't needed a public env var yet, but they come up quickly in real apps. Common use cases:

  • PUBLIC_APP_NAME — shown in the page title, footer, etc.
  • PUBLIC_API_BASE — if the client calls an external API directly
  • PUBLIC_SENTRY_DSN — client-side error tracking
  • PUBLIC_POSTHOG_KEY — analytics
<!-- src/routes/+layout.svelte -->
<script>
  import { PUBLIC_APP_NAME } from '$env/static/public';
</script>

<footer>© 2026 {PUBLIC_APP_NAME}</footer>

Because PUBLIC_APP_NAME is static, it's baked into the client bundle — no network request, no flash of missing content. If you need the value to change per deployment without rebuilding, use $env/dynamic/public instead.

Summary

Key Concepts

  • 4 modules: static/dynamic × private/public
  • PUBLIC_ prefix = opt-in to client exposure
  • Private imports blocked at build time in client code
  • $app/environment for dev/building/version
  • .env is dev-only; production uses real env vars

Rules of Thumb

  • Default to $env/dynamic/private
  • Use dev from $app/environment, not NODE_ENV
  • Declare all vars in .env for type generation
  • Keep .env.example in sync and committed
  • Never use process.env for app config
